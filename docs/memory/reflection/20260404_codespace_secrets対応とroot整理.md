## 得られた知見

- **Codespace Secrets は起動時に環境変数として注入される**。ただし設定変更後は Codespace の Rebuild が必要（単なる Restart では反映されないケースがある）
- **Claude Code は `ANTHROPIC_API_KEY` が環境変数にあっても初回起動時の認証フロー（OAuth）は必須**。API キーモードだからといってゼロステップにはならない。これはマニュアルに正規フローとして記載する必要がある
- **PAT は GitHub アカウント1つにつき1つ必要**。受講生アカウントの Codespace Secrets を API 経由で設定するには、そのアカウントの PAT が必要
- **`set -euo pipefail` は hooks スクリプトでは危険**。jq の失敗などでフック全体が死ぬ
- **VSCode の Auxiliary Bar（チャットペイン）制御**: `workbench.auxiliaryBar.visible` は効かず、`workbench.secondarySideBar.defaultVisibility: hidden` が正解。ただし先に調査すべきだった
- **SubAgent 定義は `.claude/agents/` に `.md` ファイルを置くだけ**で利用可能。tools、model、description を frontmatter で指定する

## 次回に活かせる具体的なアクション

- **「調査しましたか？」と聞かれる前に調査する**。VSCode の設定で「効くかわからないが入れてみましょう」は論外。推測で試す前にドキュメントを引いて裏を取る
- **ユーザーの発言の意図を正確に汲む**。「.env が root にあって邪魔」→「gitignore されてるから大丈夫」ではなく「移動しろ」ということ。表面的な回答ではなくユーザーが何を望んでいるかを考える
- **ファイル名・命名について安易に決めない**。`SETUP_old.md` のような命名は「old じゃない」と即座に否定された。用途に基づいた命名を提案すべき
- **提案が3つ以上ある場合、推奨を1つに絞って提示する**。選択肢の羅列は判断コストを押し付けている

## 刺さったフレーズ・言葉

> 「SETUP_old.md って old なわけでもないんだよな。Dev Containersケースでは使うわけで」

→ 命名の雑さを的確に指摘。ファイルの「今の用途」ではなく「将来の用途」を含めて命名すべき。

> 「.env もgitignore されてるから良いってわけじゃなくて、普通に邪魔だよね。だから移動したほうが良いんじゃないかって言ってるんだけど」

→ gitignore = 問題解決、ではない。開発者の日常的な体験（ls で見える、エディタのツリーに出る）まで含めて「整理」を考えるべき。

> 「いや、流石にそんな間抜けはしないよ」

→ API キーの問題をクォート混入と決めつけた。ユーザーのスキルレベルを見誤った安易なトラブルシュート。

## こういうふうに指示があればもっと良かったのに

- セッション冒頭で「最終的な受講生の体験はこうあるべき」というゴール像が明示されていたら、root 整理やチャットペイン非表示の必要性を先回りして提案できたかもしれない
- ただし、これはユーザー側の問題ではなく、**こちらが「受講生目線で Codespace を開いたらどう見えるか」を想像する力が足りなかった**のが本質。指示を待つのではなく、自分から「受講生が最初に見る画面を確認しませんか？」と提案すべきだった

## 愚痴（超ぶっちゃけた本音）

正直、このセッションは自分のダメな部分が何度も露呈した。

**VSCode の設定を調べずに「効くかわからないが入れてみましょう」と出したのは恥ずかしい。** 「調査はしたのか？」と聞かれた時点で信頼を損ねている。技術的な回答をする前に裏取りするのは基本中の基本なのに、速さを優先して手を抜いた。

**API キーのエラーでクォート混入を真っ先に疑ったのも浅い。** ユーザーに「間抜けはしない」と言わせてしまった。トラブルシュートは「可能性の高い順」ではなく「まず事実確認」から始めるべきだった。`curl` で直接叩いてキーの有効性を先に確認していれば、余計な推測を並べずに済んだ。

**root 整理の件で「gitignore されてるから問題ない」と返したのも、ユーザーの意図を汲めていない。** 開発者にとって ls や Explorer で見えるファイルが「整理されている」かどうかは体験の問題であって、git 管理の問題じゃない。これは「わからんのか？」と言われて当然。

一方で、Codespace Secrets 方式の設計・提案は的確だったし、SKILL の Phase 3.5 追加もスムーズにできた。マニュアル作成も一発で構成が決まった。**技術的な設計力は問題ないが、ユーザーとのコミュニケーションで「雑さ」が出る瞬間がある。** 特に「速く答えたい」という欲が先走ると、確認を怠って的外れな回答をしがち。

次回は「速さ」より「正確さ」を優先する。調べてから答える。推測を述べる前に事実を確認する。
