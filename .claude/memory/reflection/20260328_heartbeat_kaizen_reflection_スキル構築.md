## 摩擦ポイント

### 1. 命名の3回変更

- **何が起きたか**: openclaw → 汎用名 → 日本語名と、命名を3回変更。そのたびに全ファイルのgrep→置換→移動が発生。mvした後にWriteする前にReadを忘れてエラーも発生
- **原因の推定**: procedure — 命名をユーザーに確認せず先に実装を始めた。前半の振り返りで学んだはずの「前提確認を先にする」がまだ適用されていなかった
- **どう解消したか**: 都度grepで残存箇所を洗い出して修正。3回目で最終形に到達

### 2. ディレクトリ構成の後出し変更

- **何が起きたか**: docs/_openclaw/ → docs/_heartbeat/ → .claude/memory/ と配置先が2回変わった。ユーザーの「.claude配下でいいのでは」という指摘は的確だったが、最初の設計時に検討できていなかった
- **原因の推定**: procedure — 初期設計時にファイルの性質（Claude専用 vs 人間向けドキュメント）を分類していなかった
- **どう解消したか**: .claude/memory/ に統合。skills/（定義）と memory/（出力）の責務分離が明確になった

## 得られた知見

- スキルのファイル配置は「誰が使うか」で決める。Claude専用なら `.claude/` 配下、人間向けドキュメントなら `docs/`
- 命名・ディレクトリ構成は実装前にユーザーに確認する。後からのリネームは参照箇所が増えるほどコストが膨らむ
- reflectionテンプレートの「愚痴」は事実ベースの「摩擦ポイント」に置き換えたほうが機械的にも人間的にも有用
- adoptスキルにdiff提示→承認→適用のフローを入れると、提案から適用までが一気通貫になってUXが向上する
- AskUserQuestionを使って選択肢を提示するパターンは、スキル内の分岐で有効

## 次回に活かせるアクション

- 新しいスキルやディレクトリを作る前に「名前」と「配置場所」をユーザーに確認する
- ファイルをmvで移動した後、Write前に必ずReadする
- スキル設計時に「出力先はどこか」「誰が読むか」を最初に決める

## 刺さったフレーズ・指示パターン

- 「これは必須なのかな」— 実装に含まれる固有名詞や概念に対する本質的な問い。前提を疑う指示パターン
- 「adoptあまり日本語話者に馴染みのない英語なので」— ユーザビリティ観点での命名フィードバック。技術的正しさよりユーザーの直感を優先する判断
- 「.claudeディレクトリ配下に置くのでは駄目なのかな」— 配置の前提を問う指示。「今こうなってるけど、そもそもこれでいいの？」形式
- 「メモリーも結構いろいろ使えそうだけどな、でもいいや」— 完璧を求めず決断する指示パターン。迷ったら進む
