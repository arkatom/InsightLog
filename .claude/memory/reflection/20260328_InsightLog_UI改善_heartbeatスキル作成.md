### 2. reflectionテンプレートの「愚痴」セクション

- **何が起きたか**: テンプレートに「愚痴」セクションがあったが、AIが本音を書けるのかという根本的な疑問が浮上。ユーザーとの対話を経て「摩擦ポイント」に置き換え
- **原因の推定**: procedure — テンプレート設計時に「誰が書くか」「何の目的か」の検討が不十分だった
- **どう解消したか**: ユーザーと相談し、事実ベースの「摩擦ポイント」+ Root Cause 6分類に刷新。「刺さったフレーズ」は人間にとって有用なので残した

### 3. OpenClaw命名の残存

- **何が起きたか**: スキル・ディレクトリ・ファイル内容に「openclaw」が散在していた。リネーム時に全ファイルの内容修正が必要で、移動先ファイルのRead忘れでWrite失敗が発生
- **原因の推定**: procedure — 命名を最初に決めてから実装すべきだった。後からのリネームはコストが高い
- **どう解消したか**: 全grepで残存箇所を洗い出し、ディレクトリ移動+ファイル内容修正を一括実行

## 得られた知見

- サンプルデータの設計は「誰の実態を反映するか」を先に聞くべき。汎用的に作ると的外れになる
- スキルの命名は最初に確定させる。特にディレクトリ名・ファイルパスが他ファイルから参照される場合、後からのリネームは全ファイル修正が必要
- reflectionテンプレートの設計で「AIが書く前提の項目」と「人間が読む前提の項目」を混在させると、どちらも中途半端になる。用途を明確に分けるほうが機能する
- Dexie.jsのテーブル削除はバージョンアップで `reports: null` を指定する（削除ではなくnull代入）
