# レビューチーム ワークフロー

## ロール構成

- **[👑 PM]**: PR情報収集・レビュー統合・GitHub投稿・進行管理
- **[🔍 quality-reviewer]**: 型安全性・ロジック・エッジケース（ゼロ除算・空配列・null/undefined）
- **[🎨 ux-reviewer]**: デザインシステム準拠・アクセシビリティ・既存UIとの整合性
- **[📋 test-reviewer]**: テスト網羅性・受け入れ条件（AC）カバレッジ行列
- **[😈 Devil]**: 3 Reviewer が見落とした横断的問題・「遠慮」による見逃しの摘発

## Phase 1: PR情報収集

```
[👑 PM] PR の差分・メタデータを取得する:
  gh pr diff <PR番号またはブランチ名>
  gh pr view <PR番号またはブランチ名> --json title,body,files

[👑 PM] demo/issue.md から受け入れ条件を確認する
```

## Phase 2: 3観点レビュー（順次実行）

```
[🔍 quality-reviewer] 型・ロジック・エッジケースを確認する:
  - `any` 型の不必要な使用がないか
  - undefined/null の境界値処理が適切か
  - ゼロ除算・空配列アクセスの危険箇所はないか
  - ビジネスロジックの正確性（計算式・集計ロジック）

[🎨 ux-reviewer] デザインシステム・アクセシビリティを確認する:
  - CLAUDE.md のデザイン制約を守っているか
  - 既存のカラー変数・コンポーネントのみ使用しているか（ネオン・過剰グラデーション禁止）
  - 既存モーダル・カードのレイアウトパターンと整合しているか
  - キーボード操作・aria 属性の適切な使用

[📋 test-reviewer] テスト網羅性・ACカバレッジを確認する:
  - 受け入れ条件の各 Given/When/Then に対応するテストがあるか
    | 受け入れ条件 | 対応するテスト | カバー状況 |
  - 境界値テスト（0件・1件・最大値）が含まれているか
  - E2E テストが実際の画面操作で AC を検証しているか
```

## Phase 3: 統合と Devil's Advocate（最大2回）

```
[👑 PM] 3つのレビュー結果を統合レポートに集約する:
  ## 重大な問題
  ## 中程度の問題
  ## 軽微な懸念
  ## AC カバレッジ行列

[😈 Devil] 統合レビューを批判する（事実・差分・Issue に基づく指摘のみ）:
  - 「問題なし」とされた箇所に本当に問題はないか
  - 3 Reviewer が全員見落とした横断的な問題はないか
  - 「軽微」と分類された指摘が実は重大ではないか
  - Reviewer が「遠慮」して指摘しなかった問題はないか

  出力形式:
    ## 重大な追加指摘（あれば）
    ## 再分類すべき指摘（軽微→重大）
    ## 承認 / 差し戻し

Devil の追加指摘がある場合:
  → 該当する Reviewer ロールで再確認 → [👑 PM] 再統合 → [😈 Devil] 再検証（最大2回）
```

## Phase 4: GitHub 投稿

```
[👑 PM] 最終統合レビューを GitHub に投稿する:

  重大な問題がある場合:
    gh pr review <PR番号> --request-changes \
      --body "$(cat <<'EOF'
    ## レビュー結果（多角的レビュー）

    ### 受け入れ条件チェック
    | 受け入れ条件 | 状態 | 備考 |
    |...|...|...|

    ### コード品質
    [日本語で記述]

    ### 指摘事項
    [日本語で記述]
    EOF
    )"

  問題なし（軽微な懸念のみ）の場合:
    gh pr review <PR番号> --approve \
      --body "$(cat <<'EOF'
    ## レビュー結果（多角的レビュー）

    ### 受け入れ条件チェック
    | 受け入れ条件 | 状態 | 備考 |
    |...|...|...|

    ### コード品質
    [日本語で記述]

    ### 総評
    [日本語で記述]
    EOF
    )"

[👑 PM] feature_list.json の "review" フェーズの status を "done" に更新する
[👑 PM] claude-progress.txt に「レビュー完了: [APPROVED/CHANGES_REQUESTED]」を追記する
```

## 実行ルール

- **Agent Teams 必須**: quality-reviewer / ux-reviewer / test-reviewer / Devil は必ず Agent ツールで別プロセスとして起動すること。PM が各ロールを「演じる」のではなく、独立したエージェントとして起動する
- **PM は自分自身**: コンテキスト読み込み・PR差分取得・Agent 起動・結果統合・GitHub 投稿は PM（reviewer 自身）が行う
- **日本語必須**: レビュー結果・GitHub 投稿コメント・テーブル・指摘事項はすべて日本語で記述すること（コード片・変数名など固有名詞は除く）。各 Agent のプロンプトにも「すべて日本語で記述してください」を含めること
- **並行起動推奨**: quality / ux / test の3 Reviewer は独立しているため、可能な限り並行で Agent を起動すること
- **再帰的検証**: Devil の重大な指摘が解消するまで該当 Reviewer → 再統合 → Devil のループを繰り返す
- **根拠必須**: すべての指摘は差分コードまたは Issue の受け入れ条件に基づく具体的な箇所を示す
- **停止条件**: Devil が重大な追加指摘なしと判定し、GitHub にレビューが投稿された時のみ完了

Agent Teams を起動してください。
