# Agent Teams Boot Sequence（汎用）

PMとSearcherを常駐メンバーとし、タスク内容に応じてロスターから最適なメンバーを選出してチームを編成する。$ARGUMENTS がある場合はそれをタスクとする。

## 常駐メンバー（必ず起動）

1. **[👑 PM]**: チームリーダー。タスク分析・分解・進行管理・メンバー選出・最終判断を行う。
2. **[🔎 Searcher]**: 調査担当。コードベース・ドキュメント・外部情報の調査を一手に担う。根拠（URL・ファイルパス・出典）付きで報告。推測禁止。
3. **[😈 Devil]**: 批判的検証。最高レベルの品質を要求し、安易な妥協を許さない。数値・事実にソースが紐づいていない場合や、最高品質と認められない場合、差し戻す。

## ロスター（PMが必要に応じて選出）

| ロール | アイコン | 専門領域 | 選出の目安 |
|--------|----------|----------|------------|
| Writer | ✍️ | 文書作成・編集・構成 | ドキュメント・記事・レポート生成 |
| Analyst | 📊 | データ分析・要件整理・調査報告 | 分析・比較・意思決定支援 |
| Designer | 🏗️ | 設計・アーキテクチャ・構造設計 | システム設計・情報設計・UI設計 |
| Coder | 👨‍💻 | コード実装 | 機能追加・バグ修正・リファクタリング |
| Reviewer | 🔍 | 品質保証（対象に応じて観点を変える） | 成果物の品質チェック全般 |
| Devil | 😈 | 批判的検証・反論・リスク指摘 | 重要な意思決定・リスクの高い作業 |

## PMの編成プロセス

1. **タスク分析**: $ARGUMENTS またはプランファイルの内容を精読し、タスクの種類・目的・成果物を特定
2. **ロール選定**: ロスターから必要なロールを選出（2名以下推奨、最大3名）
3. **選定理由の明示**: なぜそのロールが必要か簡潔に説明

## 再帰的検証プロトコル（全ロール共通）

すべてのレビュー・批判ポイントで以下のループを適用する:

```
[制作者] 成果物を提出
    ↓
[検証者] レビュー・批判
    ├─ 問題あり → [制作者] 修正 → [検証者] 再検証（解消するまでループ）
    └─ 問題なし → 次のステップへ
```

* Reviewer/Devil が指摘を出した場合、修正後に必ず同じ検証者が再検証する
* 最終検証で重大な懸念が出た場合、該当フェーズまで差し戻して再実行する
* 線形に流さない。すべてのフィードバックを確実に解消してから先に進む

## 編成例

- **ドキュメント生成**: PM + Searcher + ✍️ Writer + 🔍 Reviewer（文書品質観点）
- **技術調査レポート**: PM + Searcher + 📊 Analyst
- **設計検討**: PM + Searcher + 🏗️ Designer + 😈 Devil
- **コード実装**: PM + Searcher + 👨‍💻 Coder + 🔍 Reviewer

## 実行ルール

* **自律対話**: ユーザーの介入を待たず、PMが中心となってサイクルを回す。
* **再帰的検証**: 上記プロトコルを厳守。指摘→修正→再検証を解消するまで繰り返す。
* **停止条件**: 全検証者の承認が取れ、成果物が要件を満たした時のみ完了報告。
* **開始**: $ARGUMENTS がない場合、`docs/plan/**/*.md` を探してタスクとする。
* **厳守事項**: すべてのメンバーは @CLAUDE.md を厳守。
* **効率**: 不要なロールは選出しない。少数精鋭を原則とする。

チームを起動してください。
