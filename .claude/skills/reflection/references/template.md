# セッション振り返りテンプレート

セッションごとの振り返りを記録する標準フォーマット。
人間が読んで次の指示に活かす用途と、coach（/heartbeat）が機械的に読む用途の両方を想定。

## 基本フォーマット

```markdown
# セッション振り返り - YYYYMMDD

## Handshake

1. やったこと: （1-2行で簡潔に。背景・目的も含める）
2. 結果: （成功/失敗/部分的成功）
3. 詰まり/違和感: （あれば。なければ「なし」）
4. 次回の懸念: （あれば。なければ「なし」）

## 摩擦ポイント

何に時間がかかったか、何が噛み合わなかったか。事実ベースで記述する。

- **何が起きたか**: （手戻り、エラーの繰り返し、指示の食い違い等）
- **原因の推定**: config / prompt / procedure / permission / tool / external のいずれか
- **どう解消したか**: （解消できなかった場合もそのまま書く）

## 得られた知見

- （技術的な学び、プロセスの学び、失敗から学んだこと）

## 次回に活かせるアクション

- （「こうする」「これを試す」「これは避ける」を具体的に）

## 刺さったフレーズ・指示パターン

（ユーザーの指示で効果的だった表現、うまくいったプロンプトの書き方。
　次回同じタスクで再利用できる形で記録する）
```

## 各項目の補足

### Handshake
- **振り返りの起点**。セッションで何があったかの要約
- 人間が読む「やったこと」と、coachが読む「引き継ぎメモ」を兼ねる
- 4行で収める。詳細は「摩擦ポイント」以降に書く

### 摩擦ポイント
- **改善サイクルの起点**になるセクション
- 事実を書く。感想や評価ではなく「何が起きたか」
- 原因の6分類（config / prompt / procedure / permission / tool / external）はHeartbeatのRoot Cause分類と連動。coachが機械的に読み取れるよう分類を明記する
- 摩擦がなかったセッションでは「特になし」と書いてよい

### 得られた知見
- 技術的な発見（ライブラリ、パターン、設定値）
- プロセスの発見（この順序で進めるとスムーズ等）
- 箇条書き2-5個程度

### 次回に活かせるアクション
- 抽象的な「気をつける」は禁止。具体的な行動にする
- 例: 「実装前にtsconfig.jsonのpaths設定を確認する」「UIコンポーネントは既存のsrc/components/ui/から探す」

### 刺さったフレーズ・指示パターン
- ユーザーの指示で「この言い方で一発で伝わった」という表現を記録
- うまくいったプロンプトのパターン（コンテキスト込みで）
- 次のセッションで同様のタスクがあったときに再利用する目的
